# Database migrations

SQL files here are run **manually** in the Supabase SQL Editor. They are
numbered so you run them in order; re-running is safe (every statement is
`IF NOT EXISTS` / `ON CONFLICT DO NOTHING`).

## How to run

1. Go to [supabase.com](https://supabase.com) → your project → **SQL Editor** (left sidebar)
2. Click **New query**
3. Open the file you want to run (e.g. `001_initial_schema.sql`), copy its entire contents
4. Paste into the editor
5. Click **Run** (bottom right) or `Cmd+Enter`
6. Expected result: "Success. No rows returned"

## Drift check (run this first)

`_prod_migration_check.sql` returns one row per code-critical column/table with
`present` true/false (missing objects bubble to the top). Paste it into the SQL
Editor to see exactly which migrations your live DB is missing before running
anything. (Generated by the v5 production audit; see `../AUDIT_v5.md`.)

## v5 audit catch-up migrations (2026-06-23)

Several tables/columns the code depends on were only ever documented as code
comments — never real migrations — so `db/` was not a full reproduction of prod.
These promote them to idempotent migrations. **Run any the drift check flags as
missing.** Apply-order notes (matter only for a from-scratch rebuild; on an
existing prod DB they are safe idempotent no-ops in any order):

| File | Adds | Ordering |
|------|------|----------|
| `011a_profiles_role_expires_at.sql` | `profiles.role_expires_at` | before `012`; **then re-run `012`** for the full self-update lock |
| `018b_content_ideas.sql` | `content_ideas` table + indexes | before `019` (which ALTERs it) |
| `020a_app_settings.sql` | `app_settings` table | before `021` (which UPDATEs it) |
| `024_requests_v440_columns.sql` | `requests` v4.4.0 columns + indexes | any time |
| `025_generate_log_posted.sql` | `generate_log.posted` | any time |

Also note: `018a` must run before `017` (FK on `user_id`) and `018` (merge that
reads the profile-pic / athlete_voice columns) — the original cause of the
profile-pic silent-save bug.

## After running

Visit **`/api/cloud-health`** on your deployed site (or locally) to confirm
the schema is reachable. Expected response shape:

```json
{
  "configured": true,
  "ready": true,
  "tables": { "media": { "ok": true, "rows": 0 }, ... },
  "buckets": { "media": { "ok": true }, ... }
}
```

If `ready: false`, the `notes` array tells you what's missing.
